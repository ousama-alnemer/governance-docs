📖 Policy Effect Legend — How each effect type impacts New & Existing resources ▼
🚫 Deny ▼
🆕
New resourcesBlocked immediately at deployment. The request is rejected before the resource is ever created.
📦
Existing resourcesNOT removed or changed. Marked as Non-Compliant in the Policy Dashboard. Must be fixed manually.
🔍 Audit ▼
🆕
New resourcesAllowed to be created, but immediately flagged as Non-Compliant if they violate the rule.
📦
Existing resourcesScanned every 24 hours. Flagged as Non-Compliant. No auto-fix — manual action required.
✏️ Modify ▼
🆕
New resourcesAzure automatically fixes the resource at creation — e.g. adds missing tags from the parent Resource Group.
📦
Existing resourcesMarked Non-Compliant. Run a Remediation Task to auto-fix them retroactively.
🚀 DeployIfNotExists ▼
🆕
New resourcesAzure auto-deploys the required config (e.g. diagnostic settings, shutdown schedule) within ~15 min of creation.
📦
Existing resourcesScanned every 24 h. Flagged Non-Compliant. NOT auto-fixed — must run a Remediation Task.
⚠️ Not via Policy ▼
🆕
New resourcesNo automatic enforcement. Must be configured manually as part of the onboarding checklist or landing zone setup.
📦
Existing resourcesNo scanning. No compliance tracking. Gaps must be found and fixed manually via portal, CLI, or Bicep.
