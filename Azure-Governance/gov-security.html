Network Security

What it does: Splits the network into separate subnets — one per workload layer (web, app, data). No layer can talk directly to another without going through an NSG rule.

What is enforced:
✅ Each subnet must have a dedicated purpose (e.g. snet-web, snet-app, snet-db)
✅ Subnets must use approved IP address ranges only
✅ Traffic between subnets is controlled by NSG rules
DENY How this policy affects your resources
New Subnets
❌ Blocked immediately. Deployment fails if the subnet doesn't match the approved naming or address range.
Existing VNets
⚠️ Marked Non-Compliant. Not changed automatically — IT Operations must fix during a maintenance window.
Save
What it does: Blocks NSG rules that open sensitive ports to the entire internet. Every subnet must have a firewall-style rule set attached.

What is blocked:
🚫 Inbound RDP (3389) from 0.0.0.0/0 or Any
🚫 Inbound SSH (22) from the internet
🚫 Inbound HTTP (80) from the internet
✅ Every subnet must have an NSG with a deny-all rule as the lowest priority
DENY How this policy affects your resources
New NSG Rules
❌ Blocked immediately if they open ports 22, 80, or 3389 to the internet.
Existing NSG Rules
⚠️ Flagged as Non-Compliant + high-severity in Defender for Cloud. Must be removed manually.

What it does: Forces all outbound traffic from every subnet through the Azure Firewall for inspection. No resource can talk to the internet directly.

How it works:
✅ Every subnet must have a UDR sending 0.0.0.0/0 → Azure Firewall
✅ Excluded subnets: AzureFirewallSubnet, GatewaySubnet, AzureBastionSubnet
ℹ️ This controls routing only — what the firewall allows/blocks is managed separately
DENY How this policy affects your resources
New Subnets
❌ Blocked if no UDR is attached pointing to the firewall.
Existing Subnets
⚠️ Flagged as Non-Compliant. IT Operations must add the UDR — test in dev/test before applying to production.

What it does: Removes PaaS services from the public internet. Instead of a public URL, the service gets a private IP inside your VNet — traffic never leaves Azure.

Applies to:
🔒 Key Vault | Storage Account | Service Bus | SQL Server
✅ DNS resolves the private IP via Private DNS Zones in the hub VNet
🚫 Public access (publicNetworkAccess: Enabled) is blocked in production
DENY How this policy affects your resources
New PaaS Resources
❌ Blocked if public access is enabled. Private Endpoint must be created first.
Existing PaaS Resources
⚠️ Flagged Non-Compliant. Step 1: create Private Endpoint → Step 2: verify connectivity → Step 3: disable public access.

What it does: Removes all direct VM access from the internet. The only way to connect to a VM is through Azure Bastion — a browser-based RDP/SSH session over HTTPS.

What is enforced:
🚫 No public IP addresses on VMs
🚫 No NSG rules opening port 3389 or 22 from the internet
✅ Access via Azure Portal → Bastion → RDP/SSH over port 443
✅ Standard SKU: connect by private IP, native client support, file transfer
DeployIfNotExists How this policy affects your resources
New VMs
✅ Deployed without public IP automatically. RDP/SSH from internet blocked at creation.
Existing VMs
⚠️ VMs with public IPs flagged high-severity in Defender for Cloud. Remove public IP after verifying Bastion works.

What it does: Forces App Services and Function Apps to send all outbound traffic through the VNet — not directly over the public internet.

The private data path:
App Service → snet-app subnet → Azure Firewall → Private Endpoint → PaaS service

✅ Database calls, Key Vault lookups, Service Bus — all stay private
🚫 App Services in production without VNet Integration are blocked
DENY How this policy affects your resources
New App Services
❌ Blocked in production without VNet Integration. The snet-app subnet must exist first.
Existing App Services
⚠️ Outbound traffic is public. Enable in: Networking → Outbound → VNet Integration. Requires app restart.

Save
What it does: Locks down Key Vaults so secrets, keys, and certificates are never accessible from the public internet.

What is enforced:
🚫 Public network access must be Disabled
✅ Soft delete: 90 days (recover from accidental deletion)
✅ Purge protection: enabled (prevents permanent deletion)
✅ All secret access logged to Log Analytics
DENY How this policy affects your resources
New Key Vaults
❌ Blocked if public access is on, soft delete is off, or purge protection is missing.
Existing Key Vaults
⚠️ Flagged Non-Compliant. Create Private Endpoint → verify connectivity → disable public access. No downtime needed.
